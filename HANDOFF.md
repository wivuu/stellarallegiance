   - `ClientDisconnected` — set Player.Online=false, destroy ship if any

4. **Publish** using the commands above. Use `--delete-data always` on first publish with a breaking schema:
   ```bash

       --server http://localhost:3001 --yes --delete-data always
   ```

5. **Regenerate bindings**, then `dotnet build` the client.

### Scheduling SimTick

SpacetimeDB 2.x uses a **scheduled table** pattern — not a timer API call. The `module/CLAUDE.md` has a worked example (search for "Scheduled Tables"). The pattern is:

```csharp
[SpacetimeDB.Table(Accessor="SimTickTimer", Scheduled=nameof(SimTick), ScheduledAt=nameof(ScheduledAt))]
public partial struct SimTickTimer { ... }
```

Insert one row in `Init` with `ScheduleAt.Interval(TimeSpan.FromMilliseconds(50))` (= 20 Hz). The `SimTick` reducer's signature must accept the timer struct as a second argument. **Leave the body of SimTick as a stub (`Log.Info("tick")`) for T1** — full integration is T4.

## Key facts to avoid repeating past mistakes

- The SDK builder uses **`WithDatabaseName()`** not `WithModuleName()`.
- `OnConnectError` callback signature is `Action<Exception>`, not `Action<ErrorContext>`.
- `spacetime init` puts the C# project in `module/spacetimedb/` (not `module/` directly). The `module/spacetime.json` ties it together. **Do not move these files.**
- The module's `CLAUDE.md` (generated by `spacetime init`) has accurate 2.x syntax — treat it as the ground truth for attribute names and DB operation patterns.
- The `.PLAN/` files are the design spec; they were written before the SDK was pinned and some syntax details differ. Prefer `module/CLAUDE.md` for C# specifics, `.PLAN/` for game design intent.
